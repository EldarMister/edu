# EDU POS

EDU POS - учебная POS-система для кафе и ресторана. Проект покрывает полный рабочий цикл заведения: авторизация сотрудника, выбор стола, сбор корзины, отправка заказа на кухню, real-time обработка статусов, оплата, чек, администрирование меню, залов, персонала, настроек, аудита и сверки оплат.

Проект состоит из двух приложений:

- `backend` - NestJS API с Prisma, PostgreSQL, JWT, Socket.IO, Web Push и сервисами бизнес-логики.
- `frontend` - React/Vite PWA для официанта, кухни, администратора и владельца.

## Ключевые возможности

### Общие

- Авторизация по телефону и паролю.
- JWT access/refresh token flow.
- Разделение ролей: `WAITER`, `KITCHEN`, `BAR`, `ADMIN`, `OWNER`.
- Автоматический редирект пользователя на интерфейс своей роли.
- Защищённые маршруты и backend guards по ролям.
- Глобальная обработка ошибок API.
- Валидация DTO через `class-validator`.
- Real-time обновления через Socket.IO.
- Глобальная система авто-обновления frontend (модальное окно с проверкой версий без перезагрузки кэша вручную).
- Индикатор соединения: онлайн/нет соединения.
- PWA manifest, service worker и installable-интерфейс.
- Web Push уведомления при наличии VAPID-ключей.
- Звуки уведомлений, новых заказов и оплаты.
- Toast-уведомления поверх интерфейса.
- Номера заказов отображаются в коротком виде: `№000006` -> `№6`.
- Денежные значения форматируются в сомах.
- Поддержка различных единиц измерения (`unit`: шт, мл, г) и указание времени приготовления (`cookingTime`) для каждого блюда.
- Логотип и базовый стиль EDU POS во всех интерфейсах.
- **Поддержка сетов (комбо-наборов)** с возможностью изменения состава (замена или удаление компонентов).
- **Варианты блюд** (например, объемы "0.5 л", "1 л") с разной ценой и остатками.
- **Складской учет (остатки)** с автоматическим списыванием при заказе.
- **Скидки и процент за обслуживание**, настраиваемые на уровне заведения и позиций.
- Разделение цехов (PrepStation): отправка позиций на Кухню, Бар или статус «без отправки».

### Официант

- Мобильный и desktop-интерфейс.
- Старт и завершение смены.
- Автоматическое закрытие смен, длящихся более 20 часов (фоновый cron-процесс).
- Проверка активной смены перед отправкой заказа.
- Профиль официанта с текущим статусом смены.
- Личный кабинет официанта:
  - статистика за 7 дней;
  - завершённые и отменённые заказы;
  - выручка;
  - последние заказы;
  - просмотр деталей заказа;
  - переключение языка личного кабинета `ru`/`ky`;
  - выход из аккаунта.
- Включение push-уведомлений в профиле.
- Тест звука уведомления.
- Выбор зала и стола.
- Адаптивная сетка столов.
- Цветовые статусы столов.
- Легенда статусов столов.
- Выбор блюд по категориям.
- Поиск блюд.
- Отображение недоступных блюд внизу списка меню.
- Возврат блюда на прежнее место, когда оно снова становится доступным.
- Корзина по каждому столу сохраняется отдельно при переключении столов.
- Добавление блюд в корзину.
- Выбор вариантов блюда при добавлении в корзину.
- Добавление и кастомизация сетов (удаление/замена компонентов внутри комбо-набора).
- Уменьшение количества блюда из меню.
- Изменение количества в корзине.
- Пометка блюда «С собой» (навынос) для корректной упаковки кухней/баром.
- Удаление позиции из корзины.
- Комментарий к каждой позиции.
- Общий комментарий к заказу.
- Toast после добавления блюда в формате `Название ×N добавлено`.
- Создание нового заказа.
- Добавление блюд в существующий активный заказ.
- Редактирование заказа до принятия кухней.
- Просмотр активных заказов официанта.
- Сортировка активных заказов по приоритету внимания.
- Контекстные действия в списке заказов.
- Отмена заказа с причиной.
- Обработка частичного отказа кухни:
  - продолжить без отказанной позиции;
  - выбрать замену;
  - отменить заказ.
- Действия по заказу:
  - забрать с кухни;
  - подать гостям;
  - перейти к оплате.
- Действия со столом:
  - закрыть свободный стол;
  - перенести заказ на другой стол;
  - передать стол другому официанту на смене.
- Оплата заказа:
  - QR;
  - наличные;
  - карта;
  - смешанная оплата (часть наличными, часть QR-кодом);
  - список способов оплаты берётся из настроек владельца.
- Учет процента за обслуживание и скидок в итоговой сумме.
- Показ QR-кода клиенту при QR-оплате.
- Анимированное подтверждение успешной оплаты.
- Получение и печать чека (компактный формат для термопринтеров 58-80мм).
- Возможность печати предчека (без фискальной информации).
- Направление запросов на печать чеков администратору.
- Освобождение стола после оплаты/закрытия.
- Надежная сквозная нумерация чеков, устойчивая к удалению истории.

### Кухня и Бар

Система поддерживает разделение цехов приготовления. На кухню уходит только еда, на бар — только напитки. Позиции со статусом «без отправки» (например, жвачка) никуда не отправляются, официант выдает их сам.

- Отдельные интерфейсы кухни (`/kitchen`) и бара (`/bar`).
- Доступ для ролей `KITCHEN`, `BAR`, `ADMIN`, `OWNER`.
- Real-time получение новых заказов своего цеха.
- Вкладки заказов:
  - новые;
  - в работе;
  - готовые;
  - отказанные.
- Периодический fallback-refresh заказов.
- Карточка заказа с номером, столом, временем и позициями (показываются только позиции, относящиеся к цеху).
- Пометка «С собой» у позиций навынос.
- Отображение состава сетов с независимым управлением готовностью каждого компонента.
- Просмотр комментариев к заказу и позициям.
- Принятие заказа в работу.
- Отметка заказа готовым.
- Отказ всего заказа с причиной и комментарием.
- Отказ отдельной позиции с причиной и комментарием.
- Запись kitchen events в базу.
- Уведомления официанту о готовности или отказе.
- Стоп-лист:
  - список блюд по категориям;
  - поиск блюда;
  - включение/выключение доступности;
  - Изменение стоп-листа инвалидирует меню у официанта.
- Возможность включить push-уведомления.
- Звук нового заказа и голосовая озвучка (TTS).
- Интеграция с локальным TTS-микросервисом (Silero) для автоматического прочтения вслух новых заказов.

### Администратор

Администратор видит рабочие разделы управления заведением без owner-only аналитики.

- Заказы:
  - обзор заказов;
  - список заказов с пагинацией;
  - фильтры по вкладкам;
  - поиск;
  - фильтр по датам;
  - просмотр деталей заказа;
  - отмена заказа с причиной.
- Столы и залы:
  - обзор количества залов и столов;
  - список залов;
  - создание, редактирование, удаление зала;
  - создание, редактирование, удаление стола;
  - изменение номера, посадочных мест и активности стола.
- Меню:
  - обзор количества блюд, категорий, активных блюд и средней цены;
  - список категорий с возможностью ручной сортировки (изменение порядка отображения);
  - создание и удаление категорий;
  - массовый перенос всех блюд из одной категории в другую;
  - список блюд;
  - поиск блюд;
  - фильтр по категории;
  - создание блюда;
  - редактирование блюда;
  - удаление блюда;
  - цена, описание, категория, скидка, доступность и активность.
- Персонал:
  - обзор сотрудников;
  - список сотрудников;
  - фильтр по роли;
  - поиск;
  - создание сотрудника;
  - редактирование сотрудника;
  - деактивация/удаление сотрудника;
  - отображение сотрудников на смене;
  - внесение кассовых отчетов (отметка факта сдачи наличных за смену).
- Отчеты по персоналу:
  - отчет по официантам (waiter-report): статистика выручки и заказов за день, неделю, месяц;
  - отчет по сменам (shift-report): детальная сводка по каждой смене за выбранную дату.
- Инциденты и штрафы/премии:
  - учет штрафов (penalty) и премий (reward);
  - фиксация инцидентов (например, бой посуды) с указанием суммы ущерба и фото.
- Real-time обновление заказов и столов в админке.

### Владелец

Владелец имеет все возможности администратора и дополнительные разделы.

- Статистика:
  - выручка за сегодня;
  - количество заказов;
  - средний чек;
  - выручка за выбранный период;
  - тренды;
  - график выручки;
  - способы оплаты;
  - топ блюд;
  - топ официантов;
  - периоды: сегодня, неделя, месяц, всё время, кастомный диапазон.
- Журнал действий:
  - список audit logs;
  - фильтр по датам;
  - фильтр по пользователю;
  - фильтр по типу действия;
  - подробности старого/нового значения;
  - пагинация.
- Сверка оплат (Reconciliation):
  - загрузка банковской выписки в форматах Excel (`xlsx`) или PDF (`pdf-parse`);
  - поддержка парсинга выписки через backend-алгоритм сопоставления;
  - сравнение с оплаченными POS-заказами;
  - допуск по времени 1/3/5/10 минут;
  - статусы: совпало, не найдено, требует проверки, сумма отличается, лишняя операция;
  - статистика результата сверки.
- Настройки:
  - название кафе;
  - адрес;
  - телефоны;
  - текст в чеке;
  - язык системы `ru`/`ky`;
  - включение/выключение QR, наличных и карты;
  - загрузка QR-кода для оплаты;
  - удаление QR-кода;
  - статус принтера.

## Роли и маршруты

| Роль | Маршрут | Интерфейс |
| --- | --- | --- |
| `WAITER` | `/waiter` | Официант |
| `KITCHEN` | `/kitchen` | Кухня |
| `BAR` | `/bar` | Бар |
| `ADMIN` | `/admin` | Администратор |
| `OWNER` | `/owner` | Владелец |

Маршрут `*` перенаправляет пользователя на `/login`, если он не авторизован, или на домашний маршрут его роли.

## Основные backend API

Все backend routes имеют глобальный prefix `/api`.

### Auth

- `POST /api/auth/login` - вход по телефону и паролю.
- `POST /api/auth/refresh` - обновление access/refresh токенов.
- `GET /api/auth/me` - текущий пользователь.

### Публичные справочники

- `GET /api/halls` - активные залы со столами.
- `GET /api/tables` - активные столы.
- `GET /api/categories` - активные категории.
- `GET /api/dishes` - активные блюда.
- `GET /api/settings` - публичные настройки кафе.
- `GET /api/settings/qr` - QR-код оплаты как изображение.

### Официант и заказы

- `GET /api/waiter/shifts/current` - текущая смена.
- `POST /api/waiter/shifts/start` - начать смену.
- `POST /api/waiter/shifts/end` - завершить смену.
- `POST /api/orders` - создать заказ.
- `GET /api/orders/active` - активные заказы официанта.
- `GET /api/orders/cabinet` - данные личного кабинета официанта.
- `GET /api/orders/:id` - детали заказа.
- `POST /api/orders/:id/items` - добавить позиции в заказ.
- `PATCH /api/orders/:id` - редактировать заказ до принятия кухней.
- `POST /api/orders/:id/cancel` - отменить заказ.
- `POST /api/orders/:id/picked-up` - забрать заказ с кухни.
- `POST /api/orders/:id/resolve-partial-rejection` - продолжить после частичного отказа.
- `POST /api/orders/:id/served` - отметить заказ поданным.
- `POST /api/orders/:id/to-payment` - перевести к оплате.
- `POST /api/tables/:id/close` - закрыть стол.
- `POST /api/tables/:id/move` - перенести заказ на другой стол.
- `POST /api/tables/:id/transfer` - передать стол другому официанту.
- `GET /api/tables/available-waiters` - официанты на смене для передачи стола.

### Кухня

- `GET /api/kitchen/orders?tab=new|in_work|ready|rejected` - заказы кухни по вкладке.
- `POST /api/kitchen/orders/:id/accept` - принять заказ.
- `POST /api/kitchen/orders/:id/ready` - отметить заказ готовым.
- `POST /api/kitchen/orders/:id/reject` - отказать весь заказ.
- `POST /api/kitchen/orders/:id/items/:itemId/reject` - отказать позицию.
- `GET /api/kitchen/stop-list` - получить стоп-лист.
- `PATCH /api/kitchen/stop-list` - обновить доступность блюд.

### Оплата

- `POST /api/payments` - принять оплату.
- `GET /api/payments/:orderId/receipt` - получить данные чека.

### Администратор

- `GET /api/admin/orders/overview` - сводка заказов.
- `GET /api/admin/orders` - список заказов.
- `GET /api/admin/orders/:id` - детали заказа.
- `GET /api/admin/tables/overview` - сводка столов.
- `GET /api/admin/halls` - залы.
- `POST /api/admin/halls` - создать зал.
- `PATCH /api/admin/halls/:id` - обновить зал.
- `DELETE /api/admin/halls/:id` - удалить зал.
- `POST /api/admin/tables` - создать стол.
- `PATCH /api/admin/tables/:id` - обновить стол.
- `DELETE /api/admin/tables/:id` - удалить стол.
- `GET /api/admin/menu/overview` - сводка меню.
- `GET /api/admin/categories` - категории.
- `POST /api/admin/categories` - создать категорию.
- `PATCH /api/admin/categories/:id` - обновить категорию.
- `DELETE /api/admin/categories/:id` - удалить категорию.
- `GET /api/admin/dishes` - блюда.
- `POST /api/admin/dishes` - создать блюдо.
- `PATCH /api/admin/dishes/:id` - обновить блюдо.
- `DELETE /api/admin/dishes/:id` - удалить блюдо.
- `GET /api/admin/staff/overview` - сводка персонала.
- `GET /api/admin/staff` - сотрудники.
- `POST /api/admin/staff` - создать сотрудника.
- `PATCH /api/admin/staff/:id` - обновить сотрудника.
- `DELETE /api/admin/staff/:id` - удалить/деактивировать сотрудника.

### Владелец

- `GET /api/admin/statistics` - статистика.
- `GET /api/audit-logs` - журнал действий.
- `GET /api/audit-logs/filters` - значения фильтров журнала.
- `POST /api/admin/reconciliation` - сверка оплат по загруженной выписке.
- `GET /api/admin/settings` - полные настройки.
- `PATCH /api/admin/settings` - обновить настройки.

### Push

- `GET /api/push/public-key` - статус и публичный VAPID-ключ.
- `POST /api/push/subscribe` - сохранить browser push subscription.
- `DELETE /api/push/subscribe` - удалить browser push subscription.

## Real-time события

Socket.IO используется для синхронизации интерфейсов без ручного обновления страницы.

Комнаты:

- `role:kitchen` - кухня.
- `role:admin` - админка и владелец.
- `waiter:{waiterId}` - конкретный официант.

События сервера:

- `order:new` - создан новый заказ.
- `order:status_changed` - изменился статус заказа.
- `kitchen:new_order` - новый заказ для кухни.
- `waiter:order_ready` - заказ готов для официанта.
- `waiter:order_rejected` - заказ или позиция отказаны.
- `waiter:shift_started` - смена начата.
- `waiter:shift_ended` - смена завершена.
- `table:status_changed` - изменился статус стола.
- `menu:updated` - изменилось меню или стоп-лист.
- `notification:new` - уведомление пользователю.

Frontend подключается к Socket.IO с JWT и использует `websocket` с fallback на `polling`.

## Health и Status Page

Backend имеет публичные endpoints для проверки состояния:

- `GET /api/health` - текущий backend, БД и миграции.
- `GET /api/health/migrations` - миграции текущей среды.
- `GET /api/health/project` - текущая среда плюс optional-сравнение dev/main.
- `GET /api/health/status` - простой HTML-экран со статусом backend, БД и миграций.

Status page обновляется раз в минуту и показывает ошибки подключения к БД, непримененные миграции и последние примененные migration names.

## Основной сценарий работы

1. Официант входит в систему.
2. Официант начинает смену.
3. Официант выбирает зал и стол.
4. Официант добавляет блюда в корзину.
5. Официант отправляет заказ на кухню.
6. Кухня получает заказ в real-time.
7. Кухня принимает заказ.
8. Официант видит обновление статуса.
9. Кухня отмечает заказ готовым или отказывает заказ/позицию.
10. Официант получает уведомление.
11. Официант забирает заказ с кухни.
12. Официант отмечает заказ поданным гостям.
13. Официант переводит заказ к оплате.
14. Официант выбирает способ оплаты.
15. Система фиксирует оплату.
16. Официант печатает чек.
17. Заказ закрывается, стол освобождается.

## Частичный отказ кухни

Если кухня отказала одну позицию, заказ получает статус `partially_rejected`, а официант должен принять решение:

- продолжить без отказанной позиции;
- выбрать замену и отправить её на кухню;
- отменить заказ.

Это защищает официанта от незаметного продолжения заказа с неполным составом.

## Настройки оплаты и чека

Владелец управляет настройками в разделе `/owner` -> `Настройки`.

Настройки используются:

- в окне оплаты официанта;
- при показе QR-кода клиенту;
- при генерации чека;
- для доступных способов оплаты.

Если QR включён, но QR-код не загружен, официант увидит предупреждение и не сможет подтвердить QR-оплату, пока не выбран другой способ оплаты или не загружен QR.

## Стоп-лист

Кухня управляет доступностью блюд через стоп-лист.

Логика:

- недоступные блюда остаются видимыми официанту, но отключены;
- недоступные блюда показываются ниже доступных;
- при возврате доступности блюдо возвращается в свой обычный порядок;
- изменение стоп-листа инвалидирует меню на frontend и рассылает realtime-событие.

## Аудит

В `audit_logs` пишутся важные действия:

- создание, обновление, отмена и оплата заказов;
- изменение способа оплаты;
- добавление, удаление и изменение количества позиций;
- закрытие, перенос и передача столов;
- создание, обновление и удаление блюд;
- изменение цены и доступности блюда;
- создание, обновление и удаление категорий;
- создание, обновление и удаление сотрудников;
- обновление настроек.

Журнал доступен владельцу в разделе `Журнал`.

## Сверка оплат

Сверка оплат доступна владельцу.

Backend принимает файл выписки, извлекает операции с датой и суммой, затем сравнивает их с оплаченными заказами за выбранный период. В сверку попадают банковские способы оплаты: `qr`, `card`, а также заказы без метода оплаты, если они закрыты как оплаченные.

Результат показывает:

- совпавшие платежи;
- платежи, которые не найдены в выписке;
- операции, требующие проверки;
- расхождения по сумме;
- лишние операции в выписке.

## PWA, push и звуки

Frontend собирается как PWA:

- manifest задаётся в `frontend/vite.config.ts`;
- service worker генерируется через `vite-plugin-pwa`;
- дополнительный push service worker лежит в `frontend/public/push-sw.js`;
- иконки лежат в `frontend/public/icon.png` и `frontend/public/icon1.png`.

Звуки:

- `frontend/public/sounds/notify.mp3` - обычные уведомления;
- `frontend/public/sounds/new-order.mp3` - новый заказ на кухне;
- `frontend/public/sounds/payment sound.mp3` - успешная оплата.

Web Push работает только если:

- браузер поддерживает Service Worker, PushManager и Notification API;
- пользователь дал разрешение;
- backend настроен с VAPID-ключами;
- приложение открыто в поддерживаемом контексте браузера.
