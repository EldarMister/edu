# EDU POS

EDU POS - полноценная POS/PWA-система для кафе и ресторанов с пятью рабочими контурами:

- официант;
- кухня;
- бар;
- админ/владелец;
- гости стола (QR-меню).

Проект покрывает весь операционный цикл заведения: от старта смены и создания заказа до оплаты, фискализации чека на ККМ, сверки банковских платежей, журнала действий и управления каталогом в реальном времени. Гости могут самостоятельно заказывать со своих телефонов через QR-ссылку стола — без участия официанта.


## Из чего состоит проект (полная структура)

- `backend` — NestJS API, Prisma, PostgreSQL, JWT, Socket.IO, Web Push, бизнес-логика заказов, оплаты, печати, фискализации и отчетности.
- `frontend` — React/Vite PWA с отдельными интерфейсами для официанта, кухни, бара, администратора, владельца и публичного QR-меню.
- `tts-service` — self-hosted FastAPI микросервис на Silero TTS для голосовой озвучки заказов и событий.

### Голосовая озвучка

- Self-hosted Silero TTS, без Web Speech API.
- Основная модель озвучки — `v5_2_ru`, fallback при ошибке/таймауте — `v4_ru`.
- Очередь озвучки без перебивания фраз.
- Голосовая озвучка новых заказов для кухни и бара.
- Голосовая озвучка событий официанту.
- В header кухни есть кнопка `Озвучка` с компактными настройками.
- Кухня может выбрать speaker: Baya, Ksenia, Xenia, Eugene или Aidar. `Random` не используется.
- `Уведомления` и `Голосовая озвучка` включаются отдельно: можно оставить звуковые/визуальные уведомления без TTS или наоборот.
- Скорость озвучки: `0.8x`, `1.0x`, `1.2x`, `1.5x`.
- Настройки озвучки кухни сохраняются на устройстве в localStorage: speaker, уведомления, TTS, скорость, preferred/fallback model.
- Использование `voiceName` у блюда, если нужно отдельное произношение.
- Варианты блюд тоже проговариваются: `1кг` -> `один килограмм`, `1 л` -> `один литр`, `0.5 кг` -> `пол килограмма`, `0.5 л` -> `пол литра`, `500 мл` -> `пятьсот миллилитров`.
- В озвучке учитываются номер заказа, номер стола, зал и состав заказа.
- При добавлении новых блюд в действующий заказ озвучиваются именно новые позиции.
- При отказах и заменах озвучиваются конкретные блюда, а не абстрактные сообщения.
- Замена может проговариваться в формате `старое блюдо -> новое блюдо`.

### QR-меню для гостей (EDU MENU)

Публичный интерфейс, доступный по ссылке `/menu/:tableToken` без авторизации. Каждый стол имеет неугадываемый уникальный токен, который зашит в QR-код.

- Гость сканирует QR-код стола и попадает в меню заведения.
- Меню показывает категории, карточки блюд с фото, ценами и описанием.
- Несколько гостей одного стола автоматически формируют **общий заказ** — каждый видит добавленные другими позиции в реальном времени.
- Гость идентифицируется по стабильному `guestKey` в localStorage — смена экрана и перезагрузка страницы не меняют его личность.
- После сборки корзины гости отправляют общий заказ в POS одной кнопкой.
- Созданный заказ попадает в очередь кухни и официантов с пометкой `source=qr`.
- Отправленный заказ можно проверить: экран «Мои заказы» показывает состав, общую сумму и статус прохождения через кухню в реальном времени (принят → готовится → готов → подан).
- После отправки заказа гость может добавить новые позиции — начинается второй круг с новой сессией.
- Фото блюд поддерживают как загруженные через админку изображения, так и внешние URL.
- Дисклеймер о фото на русском и кыргызском языках.
- Без участия официанта: заказ создаётся напрямую через публичный API.

### Фискализация (ККМ КР)

Интеграция с онлайн-кассами Кыргызстана. Фискальный чек пробивается автоматически при печати чека заказа.

- Поддержка провайдеров: **eKassa** и **YaKassa**.
- Mock-провайдер для тестирования интеграции без реального ключа ККМ.
- Провайдер выбирается в настройках владельца; при отсутствии выбора ККМ молча выключена.
- Фискальные данные хранятся на заказе: номер чека, фискальный признак, QR ГНС для проверки.
- При ошибке ККМ основной flow печати не блокируется — ошибка записывается в `fiscalError`.
- Идемпотентность: уже успешно пробитый чек не пробивается повторно.
- Кнопка «Повторить фискализацию» в карточке заказа для ручного исправления ошибок.
- Кнопка «Проверить соединение» с ККМ в настройках.
- Статус фискализации обновляется у администраторов через realtime (`fiscal_receipt_updated`).

## Детали системы по модулям

Этот раздел покрывает бизнес-правила, ограничения и мелкие фишки каждого модуля, которые не очевидны из общего описания.

### Заказы — бизнес-правила

- **Нумерация сбрасывается каждые сутки**: номера `№1`, `№2`, … отсчитываются в рамках `businessDate` (локальная полночь). Устойчивы к удалению истории.
- **Idempotency key**: повторный запрос создания заказа с тем же ключом возвращает уже существующий заказ, не дублирует. Применяется также для операции `add_items` (уникальный `(type, idempotencyKey)` в таблице `OrderAction`).
- **Заказ из позиций «Без отправки»** (PrepStation.none) сразу получает статус `ready`, кухня/бар его не видят.
- **Редактирование заказа** разрешено только в статусах `sent_to_kitchen`, `accepted_by_kitchen`, `cooking`. На статусах `ready`, `served`, `waiting_payment` — нельзя.
- **Перенос прогресса кухни при редактировании**: неизменные позиции сохраняют статус (`accepted`/`cooking`/`ready`) при редактировании — заказ в работе не возвращается во вкладку «Новые».
- **Отмена официантом** запрещена в статусах `waiting_payment` и `served` — только администратор/владелец.
- **Отмена оплаченного заказа** (`paid`/`cancelled`) невозможна.
- **`requiresWaiterDecision`**: блокирует переходы `kitchenReady → pickedUp → served → toPayment` пока официант не разрешил частичный отказ. Станции независимы: блокируется только та станция, у которой есть отказ.
- **Сервисный сбор** (`serviceChargeAmount`): фиксированная надбавка на весь заказ. Хранится на `Order`, обнуляется если все позиции отказаны.
- **QR-заказы** создаются с `source='qr'` и `waiterId=null`. Любой официант на смене видит такие заказы. Ограничение «один активный заказ на стол» к ним не применяется.
- **Доступ официанта к заказу** проверяется единообразно для отмены, редактирования, добавления позиций, оплаты, переноса и передачи стола. Если заказ доступен официанту по текущим правилам смены/QR, действие не должно падать с ошибкой «Это не ваш заказ».
- **Ручная корректировка статуса администратором** (`adminUpdateStatus`): разрешены `sent_to_kitchen`, `accepted_by_kitchen`, `cooking`, `ready`, `picked_up`, `served`, `waiting_payment`, `cancelled`. При отмене — запасы возвращаются; при переводе обратно в активный — вычитаются.
- **`closedAt`** на заказе проставляется при оплате, отмене и закрытии через adminUpdateStatus.

### Частичный отказ кухни — детали

- **Отказ по количеству** (`partial`): кухня отказывает не все единицы позиции — исходная позиция уменьшается, отказанная часть выделяется в отдельную строку `rejectionDecision=pending`.
- **Три решения официанта**: `removed` (убрать), `replaced` (заменить), «Принять все отказы» (bulk через `resolvePartialRejection`).
- **Замена**: новая позиция получает автокомментарий «Замена вместо [старое блюдо]» и `replacementForItemId` = id отказанной позиции.
- **Отмена готового блюда** (`cancelReadyItem`): статусы `ready`/`served`, обязательна причина. Если активных позиций не осталось — заказ переходит в `cancelled`.

### Позиции заказа — поля

- **`comment`**: на заказе в целом и отдельно на каждой позиции.
- **`takeaway`** (`true/false`): кухня видит «с собой» рядом с позицией.
- **`replacementForItemId`**: связь «чем заменили → что заменяли».

### Кухня и бар — детали

- **История «Завершённых» и «Отказанных»** хранится только 24 часа (`KITCHEN_HISTORY_WINDOW_MS`), потом исчезает из ленты.
- **Вкладка кухни** вычисляется по позициям конкретной станции независимо. Новое блюдо к уже принятому заказу не возвращает его во «Новые».
- **Пакетное «Готово»** (`kitchenReadyItems`): один вызов на несколько `itemId` + `setComponentId`. Один пересчёт статуса, одно уведомление.
- **Компоненты сета** отмечаются готовыми отдельно. Сет-родитель готов только когда готовы все компоненты.
- **Стоп-лист**: автосохранение при переключении. Откат UI при ошибке. Обновление рассылается через `MENU_UPDATED` — официанты видят изменения без перезагрузки.

### Блюда — детали

- **Фото** хранится как data URL в БД (лимит ~1.1 МБ). В ответах списка меню подменяется лёгкой ссылкой `/dishes/:id/image?v=<timestamp>`.
- **Загрузка фото по внешнему URL** (HTTPS): бэкенд скачивает изображение за 10 секунд, проверяет magic bytes (PNG/JPEG/WEBP), лимит 1 МБ, сохраняет как data URL.
- **`description`**: опциональное описание блюда.
- **`cookingTime`**: время приготовления в минутах (опциональное).
- **`isAvailable`** (≠ `isActive`): временное отключение без удаления. Используется стоп-листом. При заказе `isAvailable=false` — ошибка «Блюдо сейчас недоступно».
- **Учёт остатков** (`trackInventory + stock + initialStock + unit`): при заказе — вычет; при отмене/отказе — возврат. Поддерживается и на вариантах (`DishVariant.stock`). При нехватке — ошибка «Недостаточно остатка».
- **Рейтинг популярности** (`popularityScore`): автоматический пересчёт каждые 2 часа (cron). Формула: `сумма quantity * (1 / (1 + daysSince * 0.1))` за 90 дней, только оплаченные позиции. При оплате заказа — немедленный пересчёт задействованных блюд. При обновлении — рассылается `MENU_UPDATED`.
- **Порядок блюд в меню**: `categoryId ASC` → `popularityScore DESC` → `sortOrder ASC` → `name ASC`.
- **Компоненты сета**: каждый имеет флаги `removable` и `replaceable` — управляют тем, может ли официант убрать или заменить компонент.

### Настройки — детали

- **Singleton**: `id = "default"`, создаётся при первом обращении через upsert.
- **Минимум один способ оплаты**: нельзя отключить все три (`payQr`, `payCash`, `payCard`) одновременно — ошибка.
- **QR-оплата требует загруженного QR-кода**: при попытке принять QR-оплату без изображения — ошибка.
- **Форматы QR-кода оплаты**: только PNG, JPG, WEBP.
- **Публичные настройки** (`GET /api/public/settings`): реквизиты кафе, включённые способы оплаты, язык, флаг `fiscalEnabled` (без ключей), ссылка на QR-код.
- **QR-код оплаты** отдаётся через `/api/settings/qr?v=<timestamp>` — лёгкий endpoint вместо тяжёлого base64 в теле настроек.
- **`language`**: `ru` | `ky` — язык интерфейса, переключается у всех через realtime.
- **`instagram`, `website`**: опциональные контакты, выводятся в чеке.
- **`receiptText`**: произвольная строка на чеке (по умолчанию «Спасибо за покупку!»).
- **Аудит ключей ККМ**: `fiscalEkassaApiKey`, `fiscalYakassaApiKey` и `qrImageUrl` в журнал попадают только фактом изменения («задан»/«удалён»), сами значения не логируются.

### Оплата — детали

- **`PaymentSource`**: `normal` (обычная) или `split` (разделение счёта) — хранится на каждой строке Payment.
- **Смешанная оплата**: при вводе суммы одного способа второй автодополняется. Оба способа должны быть включены в настройках.
- **Раздельная оплата**: минимум два платежа, суммы должны совпадать с итогом (допуск 0.01). Одна строка раздельной оплаты не может быть `method=mixed`.
- **Данные чека** (`GET /api/payments/:orderId/receipt`): `cafeName`, `address`, `phone`, `phone2`, `instagram`, `website`, `receiptText`, `serviceChargeAmount`, разбивка платежей по строкам, имя официанта (или «QR-меню» для QR-заказов).
- **Звуковой сигнал при оплате**: двойной тон 740+988 Гц (`beep('payment')`). При новом заказе — `beep('newOrder')`, при принятии кухней — `beep('accept')`.
- **Анимация оплаты**: экран с галочкой показывается минимум 1.3 сек. параллельно с подгрузкой данных чека — переход к чеку только когда оба условия выполнены.

### Статистика (владелец) — детали

- **Сравнение с предыдущим периодом** (`trends`): изменение выручки, количества заказов, среднего чека в % к предыдущему аналогичному периоду.
- **Пиковые часы** (`peakHours`): топ-3 часа суток по количеству заказов (tie-break по выручке).
- **График выручки** (`revenueSeries`): `today` → по часам (24 точки); период ≤ 120 дней → по дням; период > 120 дней → по месяцам.
- **Разбивка по способам оплаты**: смешанные платежи раскладываются на наличные + QR по строкам Payment, не по `paymentMethod` заказа.
- **Топ блюд**: топ-5 по выручке за период (название + вариант, сумма, количество).
- **Топ официантов**: топ-5 по выручке (средний чек). QR-заказы группируются как «QR-меню».

### Отчёт по сменам — детали

- **Отчёт на дату** (`/admin/staff/shift-report?date=...`): оборот, наличные (должен / сдал / разница), категорийная разбивка блюд, список отменённых позиций с причинами.
- **Сдача наличных**: администратор записывает факт вручную (`/admin/staff/shift-cash`). Уникально по `waiterId + date`. Разница = сдал − должен.
- **Причины отмен** берутся из `audit_logs` (`ORDER_CANCELLED.metadata.reason`).
- **В разбивке по категориям** состав сета показывается под строкой родителя информативно, не влияет на суммы.
- **История смен** (`/admin/staff/shift-history`): периоды `today`/`week`/`month`/`custom`. Статус: `active`, `closed`, `unclosed` (активная с прошлого дня). Администратор может исправить время и закрыть смену вручную.
- **Итоги смены у официанта**: модальное окно после закрытия — начало, конец, длительность, количество заказов, сумма.

### Персонал — детали

- **Поиск** по имени (case-insensitive) и номеру телефона.
- **Нормализация телефона**: убираются нецифровые символы кроме `+` при создании и редактировании.
- **Нельзя отключить или удалить самого себя**.
- **При удалении**: если есть история заказов — мягкое удаление (`isActive=false`), если нет — физическое.
- **Управление владельцами** (роль `OWNER`): только другой владелец может видеть, создавать и менять роль `OWNER`.
- **`onShift: bool`** в списке сотрудников — активна ли смена прямо сейчас.
- **Смена обязательна** для создания заказа и добавления позиций.

### Журнал действий — полный список событий

`ORDER_CREATED`, `ORDER_CANCELLED`, `ORDER_UPDATED`, `ORDER_PAID`, `ORDER_PAYMENT_METHOD_CHANGED`, `ORDER_ITEM_ADDED`, `ORDER_ITEM_REMOVED`, `ORDER_ITEM_QUANTITY_CHANGED`, `TABLE_CLOSED`, `TABLE_MOVED`, `TABLE_TRANSFERRED`, `MENU_ITEM_CREATED`, `MENU_ITEM_UPDATED`, `MENU_ITEM_DELETED`, `MENU_ITEM_PRICE_CHANGED`, `MENU_ITEM_AVAILABILITY_CHANGED`, `CATEGORY_CREATED`, `CATEGORY_UPDATED`, `CATEGORY_DELETED`, `STAFF_CREATED`, `STAFF_UPDATED`, `STAFF_DELETED`, `SETTINGS_UPDATED`.

Журнал хранит `userName` и `userRole` на момент события — не зависит от последующего переименования сотрудника. Фильтрация: по `userId`, `actionType`, диапазону дат, с пагинацией.

### Сверка платежей — детали

- Поддерживаемые форматы банковской выписки: PDF, XLS, XLSX, CSV.
- Сопоставление по сумме и времени (допуск 3 минуты по умолчанию).
- Статусы строк: `matched`, `not_found`, `needs_review`, `amount_mismatch`, `extra`.
- Файл выписки не сохраняется на сервере.

### Склад (учёт остатков)

- Отдельная страница администратора с вкладками `Обзор`, `Блюда`, `Сырьё`, `Закупки`, `Движения`.
- **Обзор склада**: стоимость остатков, низкий остаток, закупки за период, списания сырья, динамика остатков, критичные ингредиенты, топ расхода, последние движения и топ поставщиков.
- **Блюда**: готовые товары и напитки учитываются через `Dish.stock` / `DishVariant.stock` + `trackInventory`.
- **Сырьё**: кухонные блюда списывают ингредиенты по техкартам.
- **Ручная корректировка сырья** создаёт `StockMovement` типа `correction` с источником `manual`.
- **Защита от минусового сырья** управляется настройкой `allowNegativeIngredientStock` (по умолчанию `true`): при нехватке возвращаются warnings, при `false` заказ блокируется ошибкой `Недостаточно сырья`.
- **Защита от двойного списания**: движения по заказам привязаны к `orderItemId`, одна позиция заказа списывает/возвращает конкретный ингредиент один раз.
- В карточке блюда есть предупреждение: для напитков/готовых товаров использовать остаток блюда, для кухни — техкарту; оба способа одновременно не запрещены, но показывается подсказка.

### QR-меню — детали

- **`guestKey`** генерируется при первом открытии и сохраняется в `localStorage`. При повторном открытии гость восстанавливает ту же идентичность. Стабилен через отправки заказов.
- **Снапшот состава** при отправке заказа сохраняется в `localStorage` (`edu_qr_submitted_order_<token>`) — гость видит свой заказ даже после перезагрузки страницы.
- **Экран «Мои заказы»** (`SubmittedScreen`): состав заказа (фото, название, размер, кол-во × цена, итог строки), общая сумма, трекер статуса из 4 шагов в реальном времени.
- **Статус обновляется через WebSocket** (`QR_ORDER_STATUS_CHANGED`) — официанту не нужно говорить гостю «обновите страницу».
- **«Мои заказы»** всегда доступны гостю как кнопка-FAB в нижней панели, пока заказ существует.
- **`selectedAddons`** (JSON): зарезервировано для добавок, в v1 не используется.
- **Сеты исключены из QR-меню** v1 (`isSet=true` не показываются) — состав сета настраивается только в зале.
- **`source='qr'`** на заказе + `waiterId=null`: POS понимает, что заказ пришёл с QR-меню.

### Интерфейс официанта — детали

- **Личный кабинет**: статистика за день/7 дней/месяц (выручка, завершённые, отменённые), последние 8 заказов сгруппированных по дням (Сегодня / Вчера / Ранее), кнопка «Смотреть все».
- **Переключение языка** (ru/ky) прямо в кабинете официанта и в QR-меню.
- **QR-код стола** генерируется на фронте (библиотека `qrcode`) по ссылке `<origin>/menu/<qrToken>`. Из модального окна: посмотреть, скопировать ссылку, скачать PNG, распечатать (открывается отдельное окно браузера с форматом для печати).
- **`Table.qrToken`** — UUID, не меняется при переименовании или переносе стола.
- **Предчек** (`ReceiptPrintType.preliminary`): тип запроса на печать. Помимо финального чека, официант может запросить предварительный счёт.
- **Undo-таймер** при отмене заказа: финальный запрос уходит на сервер только после задержки, дав возможность передумать.

### PWA и технические детали фронтенда

- **Обновление приложения**: SW-регистрации принудительно обновляются, затем страница перезагружается с `?v=<timestamp>` для сброса кэша. Пользователь видит модалку «Доступно обновление».
- **Звуки**: 4 типа (`notify`, `newOrder`, `payment`, `accept`). При недоступности mp3 — синтетический сигнал через WebAudio API.
- **Кэш заказов** (`order-cache.ts`): при WebSocket-событии `ORDER_STATUS_CHANGED` заказ автоматически перемещается между вкладками React Query (активные / оплаченные / отменённые) без лишних запросов к серверу.

## Важные продуктовые сценарии

### 1. Разделение счета

Система поддерживает не только обычную оплату, но и разделение заказа на несколько независимых платежей.

- каждый платеж может иметь свой способ оплаты;
- суммы платежей можно редактировать;
- один платеж может быть смешанным;
- весь заказ при этом считается `раздельной оплатой`, а не `смешанной`.

Правило отображения:

- один способ оплаты на весь заказ -> `QR-код` / `Наличные` / `Карта`;
- один платеж закрыт несколькими способами -> `Смешанная`;
- заказ закрыт несколькими платежами -> `Раздельная оплата`.

### 2. Частичный отказ кухни

Если кухня отказала не весь заказ, а только отдельные позиции:

- заказ переходит в сценарий частичного отказа;
- официант видит активные блюда отдельно от отказанных;
- решение принимается по каждой отказанной позиции отдельно;
- можно заменить позицию или убрать ее из заказа;
- итоговая сумма считается только по активным блюдам.

### 3. Печать счета и чека

Печать построена как отдельный workflow:

1. официант создает заявку;
2. заявка попадает администратору;
3. администратор подтверждает или отклоняет;
4. после реальной печати приходит событие `printed`;
5. официант получает живое уведомление без ручного обновления.

Это разделяет `заявка принята` и `документ реально распечатан`.

### 4. Real-time синхронизация без перезахода

Изменения владельца и администратора сразу отражаются у других ролей:

- обновление настроек оплаты;
- загрузка и удаление QR-кода;
- изменение категорий и блюд;
- изменение стоп-листа;
- создание, удаление и редактирование столов и залов;
- изменения меню, сетов и вариантов;
- входящие заявки на печать и статусы печати;
- обновления заказов и статусов столов.

## Что важно помнить при разработке

- История заказов хранится по snapshot-полям, поэтому блюда и категории можно удалять без поломки старых заказов.
- Изменения меню, стоп-листа, настроек, столов и залов должны проверяться сразу в нескольких ролях, потому что проект сильно завязан на realtime.
- `voiceName` у блюда влияет на озвучку кухни и официанта.
- `PrepStation.none` означает, что позиция не должна попадать на кухню или в бар.
- QR-оплата зависит не только от toggle, но и от загруженного QR-изображения.
- Автосохранение owner settings работает без отдельной кнопки подтверждения.
- Печатный поток делит статусы `approved` и `printed`, это нельзя смешивать в UI и логике уведомлений.
- QR-меню работает полностью без JWT: гость идентифицируется только через `guestKey` в localStorage. `guestKey` стабилен — он переживает отправку заказа и начало нового круга.
- После отправки общего заказа (`submitted`) draft-сессии нет. При следующем `addItem` создаётся новая сессия автоматически через `ensureSessionGuest` — это нормальный сценарий второго заказа.
- Фискализация не блокирует печать: ошибка ККМ записывается в `Order.fiscalError` и не кидает исключение в основной flow. Администратор видит бейдж и может нажать «Повторить».
- `fiscalProvider=null` в настройках — ККМ молча выключена, никаких запросов к провайдеру не отправляется.
- Telegram backup-бот проверяет `chat_id` по whitelist и требует `TELEGRAM_WEBHOOK_SECRET`. Без этих переменных webhook принимает любые сообщения от Telegram, но `/backup` не выполнится.
- Нельзя отключить все способы оплаты одновременно: всегда должен остаться хотя бы один.
- `popularityScore` пересчитывается автоматически, вручную влиять на него нельзя — только через реальные продажи.
- `Table.qrToken` не меняется никогда: напечатанные QR-коды столов остаются рабочими даже после переименования зала или изменения номера стола.
- Учёт остатков (`trackInventory`) работает независимо от `isAvailable` — блюдо может быть доступным, но с нулевым остатком заказать нельзя.
- Ключи ККМ в журнале аудита не логируются в открытом виде — только факт изменения.
