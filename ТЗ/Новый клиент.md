# ТЗ: Выпуск мобильного приложения EDU POS на React Native без отключения текущей PWA

## 1. Цель

Необходимо выпустить мобильное приложение EDU POS для телефонов на React Native.

Важно: текущая PWA-версия проекта не заменяется и не отключается.

После выпуска должны одновременно работать:

* текущая PWA в браузере на компьютере;
* текущая PWA в браузере на телефоне;
* новое мобильное приложение React Native на телефоне.

Все клиенты должны работать через один существующий backend, одну базу данных PostgreSQL и один realtime-слой Socket.IO.

React Native приложение не должно создавать отдельную POS-систему. Оно является новым клиентом к уже существующей системе EDU POS.

---

## 2. Текущий проект

EDU POS уже реализован как полноценная POS/PWA-система для кафе и ресторанов.

В проекте уже существуют:

* backend на NestJS;
* база PostgreSQL через Prisma;
* JWT авторизация;
* Socket.IO realtime;
* PWA frontend на React/Vite;
* интерфейсы официанта, кухни, бара, администратора, владельца;
* публичное QR-меню для гостей;
* Web Push;
* TTS service на FastAPI/Silero;
* оплаты, чеки, фискализация, склад, отчеты, аудит и другие существующие модули.

Задача React Native приложения — использовать уже существующую бизнес-логику, а не переписывать её заново.

---

## 3. Главный принцип архитектуры

React Native приложение должно быть отдельным клиентом к существующему backend.

```text
React/Vite PWA       ─┐
                      ├── Existing NestJS Backend ── PostgreSQL
React Native Mobile  ─┘
```

Общими остаются:

* backend API;
* Socket.IO события;
* база данных;
* пользователи;
* роли;
* заказы;
* столы;
* смены;
* меню;
* кухня;
* бар;
* настройки;
* платежи;
* уведомления;
* realtime-синхронизация.

---

## 4. Что НЕ нужно делать

В рамках этого ТЗ не нужно заново реализовывать существующую POS-логику.

Не нужно заново делать:

* модуль заказов;
* модуль кухни;
* модуль бара;
* модуль оплаты;
* модуль чеков;
* модуль фискализации;
* модуль склада;
* модуль QR-меню;
* модуль персонала;
* модуль статистики;
* модуль TTS;
* существующие Socket.IO события;
* существующие backend routes.

Нельзя:

* отключать PWA;
* ломать текущий frontend;
* менять существующую бизнес-логику без необходимости;
* делать отдельный backend для React Native;
* делать отдельную базу данных;
* дублировать сущности заказов, столов, пользователей или смен;
* делать несовместимые статусы заказов между PWA и mobile;
* переписывать backend только ради мобильного приложения.

---

## 5. Что нужно сделать

Нужно создать новый React Native клиент, который:

* авторизуется через существующий `/api/auth/login`;
* использует существующий JWT access/refresh flow;
* получает данные через существующие REST API;
* подключается к существующему Socket.IO;
* отображает актуальные данные из общей базы;
* реагирует на realtime-события так же, как PWA;
* отправляет действия пользователя в существующий backend;
* работает параллельно с PWA без конфликтов.

---

## 6. Синхронизация между PWA и React Native

Все действия должны синхронизироваться между клиентами.

### Пример 1

Официант открыл стол и создал заказ в React Native.

В PWA должны сразу обновиться:

* статус стола;
* список активных заказов;
* кухня или бар должны получить заказ, если позиции относятся к ним.

### Пример 2

Кухня в PWA приняла заказ или отметила блюдо готовым.

React Native приложение официанта должно получить:

* realtime-событие;
* обновление статуса заказа;
* уведомление;
* актуальный статус стола.

### Пример 3

Админ или владелец изменил меню в PWA.

React Native должен получить актуальное меню после обновления данных или через событие `menu:updated`.

### Пример 4

Официант работает через PWA на планшете, другой официант через React Native на телефоне.

Оба должны видеть одинаковые статусы столов, заказов и смен.

---

## 7. Scope первой версии React Native

Первая версия React Native приложения должна быть направлена на рабочие мобильные сценарии.

### Обязательно в v1

Роли:

* WAITER;
* KITCHEN;
* BAR.

Экраны:

* Login;
* восстановление сессии;
* экран официанта;
* экран кухни;
* экран бара;
* профиль пользователя;
* состояние смены;
* подключение Socket.IO;
* обработка уведомлений;
* состояние online/offline.

### Желательно в v1, если не усложняет релиз

* базовый просмотр активных заказов для администратора;
* базовые уведомления администратора;
* возможность быстро перейти в PWA/admin через браузер или WebView.

### Не обязательно в v1

Полноценные мобильные интерфейсы:

* Owner dashboard;
* глубокая аналитика;
* склад;
* сверка платежей;
* управление меню;
* управление персоналом;
* ККМ-настройки;
* audit logs.

Эти модули уже есть в PWA и могут оставаться там. Для администратора и владельца PWA остается основным рабочим интерфейсом.

---

## 8. QR-меню гостей

QR-меню уже реализовано как публичный web-интерфейс `/menu/:tableToken`.

В рамках React Native приложения не нужно переносить QR-меню в мобильное приложение.

Гости не должны устанавливать приложение.

QR-меню остается работать через браузер, как сейчас.

React Native приложение должно только корректно получать заказы, которые пришли из QR-меню, через существующий backend и Socket.IO.

---

## 9. Авторизация

React Native должен использовать текущую авторизацию.

Требования:

* login через существующий backend;
* хранение access/refresh token в защищенном хранилище;
* автоматическая подстановка access token в Axios;
* refresh token при истечении access token;
* logout;
* редирект в интерфейс по роли;
* защита экранов по роли;
* корректная обработка ошибки 401.

Рекомендуемые библиотеки:

* `@react-native-async-storage/async-storage` или secure storage;
* `axios`;
* `zustand`;
* `@tanstack/react-query`.

---

## 10. Socket.IO

React Native должен подключаться к существующему Socket.IO серверу.

Нужно использовать те же события, что уже используются в PWA.

Приложение должно обрабатывать минимум:

* `order:new`;
* `order:status_changed`;
* `kitchen:new_order`;
* `waiter:order_ready`;
* `waiter:order_rejected`;
* `waiter:shift_started`;
* `waiter:shift_ended`;
* `table:status_changed`;
* `tables:updated`;
* `menu:updated`;
* `settings:updated`;
* `notification:new`;
* события печати, если они нужны на мобильном интерфейсе.

Если текущая логика Socket.IO уже авторизует клиента по JWT, React Native должен подключаться тем же способом.

Если для мобильного клиента нужно добавить `deviceId`, это должно быть расширением, не ломающим PWA.

---

## 11. Backend-изменения

Backend не переписывается.

Разрешены только точечные доработки, если они нужны для мобильного клиента.

Допустимые backend-задачи:

1. Проверить, что текущие REST API удобно использовать из React Native.
2. Проверить CORS / origin / mobile access.
3. Проверить авторизацию Socket.IO из мобильного клиента.
4. Добавить регистрацию мобильного устройства, если нужны native push notifications.
5. Добавить хранение mobile device token.
6. Добавить отправку native push notifications.
7. Добавить отдельные lightweight endpoints только если существующие ответы слишком тяжелые для мобильного клиента.
8. Добавить версионирование mobile app, если нужно отслеживать устаревшую версию приложения.

Пример допустимой новой сущности:

```prisma
model UserDevice {
  id           String   @id @default(uuid())
  userId       String
  platform     String
  deviceId     String?
  pushToken    String?
  appVersion   String?
  isActive     Boolean  @default(true)
  createdAt    DateTime @default(now())
  updatedAt    DateTime @updatedAt
}
```

Эта сущность нужна только для мобильных push-уведомлений и не должна ломать текущий `PushSubscription` для PWA.

---

## 12. Push-уведомления

PWA продолжает использовать существующий Web Push.

React Native должен использовать native push notifications.

Важно разделить:

* Web Push для PWA;
* Native Push для React Native.

Backend должен понимать тип устройства:

```text
web-pwa
android
ios
```

Push-уведомления в React Native нужны для:

* нового заказа на кухню;
* нового заказа в бар;
* готовности заказа;
* отказа позиции;
* важных уведомлений официанту;
* будущей рации.

Если приложение открыто, основное обновление должно приходить через Socket.IO.

Если приложение свернуто или закрыто, должен приходить push.

* белый фон;
* синий основной цвет EDU POS;
* зеленый индикатор online / active;
* карточки с мягкими границами;
* крупные touch-friendly кнопки;
* нижняя навигация для рабочих ролей;

## 16. Роль WAITER в React Native

Нужно реализовать мобильный интерфейс официанта, который использует существующие API и существующую бизнес-логику.

Минимально:

* старт смены;
* завершение смены;
* профиль официанта;
* выбор зала;
* выбор стола;
* отображение статусов столов;
* блокировка чужих занятых столов по существующим правилам backend;
* просмотр меню;
* поиск по меню;
* категории;
* добавление блюда в корзину;
* варианты блюда;
* сеты, если они уже приходят через API;
* комментарии к позиции;
* комментарий к заказу;
* пометка `С собой`;
* создание заказа;
* добавление блюд в существующий заказ;
* просмотр активных заказов;
* перевод заказа по статусам через существующие endpoints;
* обработка частичного отказа кухни;
* переход к оплате;
* создание заявки на печать счета/чека, если этот сценарий нужен официанту в mobile.

Важно: все правила редактирования, отмены, частичного отказа, передачи и закрытия стола должны оставаться на backend. Mobile только вызывает существующие endpoints и показывает результат.

## 17. Роль KITCHEN в React Native

Нужно реализовать мобильный интерфейс кухни.

Минимально:

* список новых заказов;
* список заказов в работе;
* готовые;
* отказанные;
* принятие заказа в работу;
* готовность выбранных позиций;
* отказ позиции с причиной;
* отказ заказа, если это уже предусмотрено backend;
* комментарии к заказу и позициям;
* отображение `С собой`;
* поддержка компонентов сета;
* realtime-получение новых заказов;
* звук нового заказа.

TTS service уже существует. В рамках React Native v1 не нужно переносить TTS внутрь приложения. Достаточно корректно обрабатывать уведомления и звук на мобильном клиенте.

## 20. Offline / плохой интернет

Полноценный offline-режим в v1 не требуется.

Но обязательно нужно:

* показывать online/offline состояние;
* показывать reconnect Socket.IO;
* блокировать критические действия без интернета;
* повторять безопасные GET-запросы после восстановления соединения;
* не создавать дубли заказов.

Создание заказов должно использовать существующий idempotency key, если он уже применяется в backend.

---

## 21. Кэширование данных

React Native должен использовать TanStack Query для серверных данных.

Рекомендации:

* меню можно кэшировать;
* категории можно кэшировать;
* столы и заказы должны обновляться через Socket.IO events + refetch при необходимости;
* после событий `menu:updated`, `tables:updated`, `order:status_changed` нужно инвалидировать соответствующие queries;
* при reconnect нужно refetch активных данных.

---

## 22. Типы и DTO

Желательно не дублировать вручную типы, если можно переиспользовать или сгенерировать общие DTO.

Допустимые варианты:

1. создать общий пакет `shared`;
2. вынести только безопасные frontend-типы;
3. временно создать типы в `mobile/src/types`, но следить за совместимостью с backend.

Не нужно ради этого делать большой рефакторинг backend и PWA.

---

## 23. Рация

Рация не входит в первый релиз React Native приложения.

Референс рации сохранен в папке `designe/`.

Рация должна быть отдельным будущим этапом после выпуска базового мобильного приложения.

Причина:

* сначала нужно стабильно выпустить мобильный клиент;
* проверить авторизацию, заказы, realtime, push;
* после этого добавлять микрофон, аудио и push-to-talk.

Будущая рация должна работать через единый backend и связывать пользователей разных клиентов.

Планируемые каналы:

* Все;
* Официанты;
* Кухня;
* Администрация.

Но в рамках текущего ТЗ рацию не реализовывать.

---

## 24. Совместимость с PWA

После добавления React Native:

* PWA должна работать как раньше;
* маршруты PWA не меняются;
* PWA service worker не ломается;
* Web Push PWA не ломается;
* QR-меню остается публичным web-интерфейсом;
* backend API сохраняет обратную совместимость;
* текущие пользователи могут продолжить работать через браузер.

* официант в PWA → кухня в React Native;
* кухня в PWA → официант в React Native;

## 26. Критерии готовности

Задача считается готовой, если:

* PWA продолжает работать без изменений;
* React Native приложение авторизуется через существующий backend;
* роли WAITER, KITCHEN и BAR открывают свои интерфейсы;
* заказы, созданные в React Native, видны в PWA;
* заказы, созданные в PWA, видны в React Native;
* статусы заказов синхронизируются через Socket.IO;
* статусы столов одинаковые в PWA и React Native;
* кухня и бар получают новые заказы в реальном времени;
* официант получает уведомления о готовности/отказе;
* приложение корректно переживает reconnect;
* push-уведомления работают на мобильном устройстве;
* нет отдельной базы;
* нет отдельного backend;
* QR-меню продолжает работать через браузер;
* текущие рабочие сценарии PWA не сломаны.
