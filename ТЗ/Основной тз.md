# ТЕХНИЧЕСКОЕ ЗАДАНИЕ

## POS-система для кафе/ресторана

## 1. Цель проекта

Необходимо разработать современную POS-систему для кафе/ресторана, где официанты смогут принимать заказы со смартфона, кухня будет получать заказы в реальном времени, администратор сможет управлять процессами, а владелец — видеть статистику и контролировать бизнес.

Система должна работать без перезагрузки страницы, в реальном времени. Проект должен быть адаптирован под телефоны и компьютеры, с небольшим уклоном на мобильную версию, так как официанты будут работать в основном с телефона.

Проект должен быть разработан сразу с учётом production-ready подхода: нормальная архитектура, роли, безопасность, база данных, real-time события, логирование и возможность дальнейшего расширения.

# 4. Официантская часть

## 4.1 Главная логика официанта

Официант работает в основном с телефона. На одном экране должны быть:

* выбор зала;
* список столов;
* статус каждого стола;
* выбранный стол;
* меню блюд;
* корзина заказа;
* кнопка отправки заказа на кухню.

Официант выбирает стол, затем выбирает блюда для этого стола и отправляет заказ на кухню.

## 4.2 Статусы столов

У каждого стола должны быть статусы:

* свободен;
* занят;
* заказ отправлен на кухню;
* кухня приняла заказ;
* кухня отказала по части заказа или полностью;
* заказ готов;
* заказ вынесен;
* ожидание оплаты;
* оплачен.

Цвета статусов должны быть понятными:

* свободен — светлый/зелёный;
* занят — жёлтый;
* готовится — оранжевый;
* готов — синий или зелёный;
* отказ — красный;
* оплачен — серый/нейтральный.

## 4.3 Создание заказа

Официант должен иметь возможность:

* выбрать стол;
* открыть новый заказ;
* выбрать категорию блюд;
* выбрать блюдо;
* добавить количество;
* уменьшить количество;
* удалить блюдо из заказа;
* добавить комментарий к блюду;
* добавить общий комментарий к заказу;
* отправить заказ на кухню.

Пример комментария к блюду:

* без лука;
* острый;
* без майонеза;
* подать позже.

## 4.4 Отправка на кухню

После нажатия кнопки «Отправить на кухню»:

* заказ появляется на кухонном экране в реальном времени;
* официант видит статус «Отправлено на кухню»;
* заказ нельзя полностью удалить без прав администратора;
* можно добавить дополнительный заказ к тому же столу.

## 4.5 Уведомления официанту

Официанту должны приходить уведомления:

* кухня приняла заказ;
* кухня отказала в заказе или блюде;
* заказ готов;
* необходимо забрать заказ;
* заказ изменил статус.

Пример уведомления:

> Стол №5 — заказ готов. Заберите с кухни.

## 4.6 Вынесение заказа

После того как кухня отметила заказ как готовый, официант должен нажать:

* «Забрал с кухни»;
* «Вынес гостям».

После этого статус заказа меняется на «Вынесен».

## 4.7 Оплата

После того как гости поели, официант переходит к оплате.

На экране оплаты должно быть:

* номер стола;
* список блюд;
* количество;
* цена;
* скидка, если есть;
* итоговая сумма;
* выбор способа оплаты:

  * QR;
  * наличка;
  * карта — можно добавить позже;
* кнопка «Принять оплату»;
* кнопка «Печать чека».

После оплаты заказ закрывается, а стол становится свободным.

## 4.8 Чек

После оплаты официант должен иметь возможность распечатать чек.

Чек должен содержать:

* название кафе;
* дата и время;
* номер заказа;
* номер стола;
* официант;
* список блюд;
* количество;
* цена;
* скидка;
* итог;
* способ оплаты;
* текст благодарности.

# 5. Кухонная часть

## 5.1 Экран кухни

Кухня работает с планшета или компьютера.

На кухонном экране должны отображаться новые заказы в реальном времени.

Карточка заказа должна содержать:

* номер заказа;
* номер стола;
* имя официанта;
* время создания;
* список блюд;
* количество;
* комментарии к блюдам;
* общий комментарий;
* статус заказа.

## 5.2 Принятие заказа

Кухня должна иметь кнопку:

* «Принять заказ».

После принятия заказа:

* официант получает уведомление;
* статус заказа меняется на «Кухня приняла»;
* заказ переходит в работу.

## 5.3 Отказ от заказа

Если нет ингредиентов или есть другая причина, кухня должна нажать:

* «Отказать».

После этого кухня выбирает причину отказа:

* нет ингредиентов;
* блюдо временно недоступно;
* большая загруженность;
* ошибка в заказе;
* другая причина.

Если выбрана «другая причина», должно быть поле для комментария.

Важно: отказ может быть:

* по всему заказу;
* по отдельному блюду.

Пример:

> Стол №4. Отказано: Пицца Пепперони. Причина: нет сыра.

Официант должен сразу получить уведомление об отказе.

## 5.4 Готовность заказа

После приготовления кухня нажимает:

* «Готово».

После этого:

* официанту приходит уведомление;
* стол получает статус «Заказ готов»;
* заказ переходит в список готовых.

## 5.5 Статусы кухни

У заказа на кухне должны быть статусы:

* новый;
* принят;
* готовится;
* готов;
* отказан;
* частично отказан;
* выдан официанту.

# 6. Административная часть

Администратор работает в основном с компьютера, но интерфейс также должен быть адаптирован под телефон.

## 6.1 Главный экран администратора

Администратор должен видеть в реальном времени:

* активные заказы;
* новые заказы;
* заказы в работе;
* готовые заказы;
* оплаченные заказы;
* отказанные заказы;
* какие столы заняты;
* какие официанты работают;
* кто сколько заказов обслужил;
* общую сумму за сегодня;
* последние инциденты.

Интерфейс не должен быть перегруженным. На главном экране должны быть только самые важные показатели.

## 6.2 Заказы

Администратор должен видеть:

* все текущие заказы;
* прошлые заказы;
* заказ по номеру;
* заказ по столу;
* заказ по официанту;
* заказ по дате;
* заказ по статусу;
* способ оплаты;
* итоговую сумму;
* скидку;
* историю изменения заказа.

Администратор должен иметь возможность:

* открыть заказ;
* посмотреть детали;
* отменить заказ;
* изменить статус заказа;
* добавить комментарий;
* сделать скидку;
* распечатать чек повторно.

## 6.3 Управление меню

Администратор должен иметь возможность:

* добавлять блюда;
* редактировать блюда;
* удалять блюда;
* временно отключать блюдо;
* менять цену;
* менять фото блюда;
* менять описание;
* менять категорию;
* добавлять скидку на блюдо;
* убирать скидку;
* менять порядок отображения блюд.

У блюда должны быть поля:

* название;
* категория;
* описание;
* цена;
* фото;
* статус: активно/неактивно;
* скидка;
* итоговая цена со скидкой;
* время приготовления, если нужно;
* дата создания;
* дата обновления.

## 6.4 Категории блюд

Администратор должен иметь возможность:

* добавлять категории;
* редактировать категории;
* удалять категории;
* менять порядок категорий;
* включать/выключать категорию.

Примеры категорий:

* салаты;
* первые блюда;
* вторые блюда;
* роллы;
* пицца;
* напитки;
* десерты.

## 6.5 Управление столами и залами

Администратор должен иметь возможность управлять залами и столами.

### Залы

Функции:

* добавить зал;
* изменить название зала;
* удалить зал;
* включить/отключить зал.

Пример:

* Основной зал;
* Летняя зона;
* VIP-зал;
* Второй этаж.

### Столы

Функции:

* добавить стол;
* изменить номер стола;
* указать зал;
* указать количество мест;
* удалить стол;
* временно отключить стол;
* изменить порядок отображения.

У стола должны быть поля:

* номер стола;
* зал;
* количество мест;
* статус;
* активен/неактивен.

## 6.6 Управление персоналом

Администратор должен иметь возможность:

* добавлять сотрудника;
* редактировать сотрудника;
* отключать сотрудника;
* менять роль сотрудника;
* менять пароль;
* видеть активность сотрудника;
* видеть количество обслуженных заказов;
* видеть сумму заказов официанта;
* видеть рабочую смену.

Роли персонала:

* официант;
* кухня;
* администратор;
* владелец.

## 6.7 Штрафы и награды

Администратор должен иметь возможность назначить официанту или сотруднику:

* штраф;
* награду/бонус.

Поля штрафа/награды:

* сотрудник;
* тип: штраф или бонус;
* сумма;
* причина;
* дата;
* кто назначил.

Примеры причин штрафа:

* опоздание;
* ошибка в заказе;
* жалоба клиента;
* разбитая посуда;
* нарушение правил.

Примеры причин награды:

* хорошее обслуживание;
* высокая выручка;
* положительный отзыв;
* помощь команде.

## 6.8 Инциденты

В системе должна быть кнопка «Инцидент».

Администратор или официант может создать инцидент.

Примеры инцидентов:

* гости разбили посуду;
* официант разбил посуду;
* конфликт с гостем;
* жалоба;
* ошибка в заказе;
* возврат блюда;
* техническая проблема.

Поля инцидента:

* тип инцидента;
* описание;
* стол;
* заказ, если связан;
* сотрудник, если связан;
* сумма ущерба, если есть;
* фото, если нужно;
* статус:

  * новый;
  * в обработке;
  * закрыт;
* кто создал;
* дата и время.

# 7. Владелец

Владелец имеет все возможности администратора, но дополнительно видит расширенную статистику.

Интерфейс владельца не должен быть слишком перегруженным. Статистика должна быть простой, понятной и полезной.

## 7.1 Главная статистика владельца

Владелец должен видеть:

* выручка за сегодня;
* выручка за неделю;
* выручка за месяц;
* количество заказов;
* средний чек;
* самые продаваемые блюда;
* самые слабые блюда;
* лучшие официанты;
* способы оплаты;
* скидки;
* отмены;
* отказы кухни;
* инциденты.

## 7.2 Аналитика блюд

Владелец должен видеть:

* какие блюда продаются больше всего;
* какие блюда продаются меньше всего;
* общая выручка по каждому блюду;
* количество продаж по каждому блюду;
* блюда с частыми отказами;
* блюда со скидками.

## 7.3 Аналитика официантов

Владелец должен видеть:

* сколько заказов обслужил официант;
* сумма продаж официанта;
* средний чек официанта;
* количество отмен;
* количество инцидентов;
* штрафы;
* бонусы.

## 7.4 Финансовая статистика

Владелец должен видеть:

* общую выручку;
* выручку по способу оплаты:

  * QR;
  * наличка;
  * карта, если будет добавлена;
* скидки;
* возвраты;
* отмены;
* чистую сумму после скидок.

# 8. Real-time логика

Система должна работать в реальном времени без обновления страницы.

Real-time события:

* новый заказ отправлен на кухню;
* кухня приняла заказ;
* кухня отказала;
* кухня отметила заказ как готовый;
* официант забрал заказ;
* официант вынес заказ;
* заказ оплачен;
* стол изменил статус;
* администратор изменил меню;
* администратор отключил блюдо;
* создан инцидент;
* обновлена статистика.

Для этого использовать:

* Socket.IO или WebSocket;
* Redis Pub/Sub, если понадобится масштабирование;
* optimistic UI на frontend, где уместно.

# 9. PWA и мобильная версия

Проект должен работать как PWA.

Требования:

* возможность установить приложение на телефон;
* адаптация под Android;
* адаптация под iPhone;
* быстрый запуск;
* иконка приложения;
* splash screen;
* responsive layout;
* удобные крупные кнопки;
* нижняя навигация для телефона;
* работа без лишних перезагрузок;
* сохранение авторизации.

Важно: offline-режим для заказов в MVP можно не делать, потому что заказы должны идти в реальном времени. Но система должна красиво показывать ошибку, если интернет пропал.

При потере соединения:

* показать статус «Нет соединения»;
* запретить отправку нового заказа;
* автоматически переподключиться;
* после восстановления соединения обновить данные.

# 12. Авторизация и безопасность

## Авторизация

Пользователь входит по:

* номеру телефона или логину;
* паролю или PIN-коду.

Для официантов и кухни лучше сделать быстрый вход по PIN-коду, но с безопасной серверной проверкой.

## Роли и доступы

Каждая роль должна видеть только свои функции.

Официант не должен видеть:

* общую выручку;
* управление меню;
* управление персоналом;
* статистику владельца.

Кухня не должна видеть:

* оплату;
* финансы;
* персонал;
* управление столами.

Администратор не должен видеть некоторые расширенные финансовые данные, если владелец ограничит доступ.

Владелец видит всё.

## Безопасность

* JWT access token;
* refresh token;
* хэширование паролей через bcrypt;
* защита API по ролям;
* валидация данных;
* защита от случайного удаления;
* audit log для важных действий.

# 14. Статусы заказов

Заказ должен иметь понятные статусы:

* draft — черновик;
* sent_to_kitchen — отправлен на кухню;
* accepted_by_kitchen — принят кухней;
* cooking — готовится;
* ready — готов;
* picked_up — забран официантом;
* served — вынесен гостям;
* waiting_payment — ожидание оплаты;
* paid — оплачен;
* rejected — отказан;
* partially_rejected — частично отказан;
* cancelled — отменён.

---

# 15. Статусы блюда внутри заказа

Каждое блюдо внутри заказа может иметь отдельный статус:

* new;
* accepted;
* cooking;
* ready;
* rejected;
* served;
* cancelled.

Это важно, потому что кухня может отказать не во всём заказе, а только в одном блюде.

---

# 16. Печать чеков

Система должна поддерживать печать чека.

Варианты реализации:

1. Через браузерную печать.
2. Через локальный print-agent.
3. Через ESC/POS термопринтер.

Для MVP можно начать с браузерной печати, но архитектуру сделать так, чтобы потом подключить ESC/POS принтер.

Чек должен быть компактным и подходить для термопринтера.

# 17. Уведомления

Уведомления должны быть:

* внутри системы;
* звуковые уведомления для кухни;

* push-уведомления PWA, если возможно.

* новый заказ;

* заказ отказан;

# 26. Пример пользовательского сценария

## Сценарий заказа

1. Официант входит в систему.
2. Выбирает зал.
3. Выбирает стол №5.
4. Добавляет блюда:

   * Пицца Пепперони — 1 шт.
   * Цезарь ролл — 2 шт.
   * Кола — 3 шт.
5. Добавляет комментарий: «Пиццу без острого».
6. Нажимает «Отправить на кухню».
7. Кухня видит новый заказ.
8. Кухня нажимает «Принять».
9. Официант получает уведомление: «Заказ стола №5 принят кухней».
10. Кухня готовит заказ.
11. Кухня нажимает «Готово».
12. Официант получает уведомление: «Стол №5 — заказ готов».
13. Официант нажимает «Забрал с кухни».
14. Официант нажимает «Вынес гостям».
15. После еды официант открывает оплату.
16. Выбирает способ оплаты: QR или наличка.
17. Нажимает «Принять оплату».
18. Печатает чек.
19. Заказ закрывается.
20. Стол становится свободным.
