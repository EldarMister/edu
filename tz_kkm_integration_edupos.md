# Задача: Интеграция ККМ в edu-pos (Кыргызстан)

## Контекст проекта

Ты работаешь с существующей POS-системой **edu-pos** для кафе/ресторанов в Кыргызстане.
Система уже в production на реальных заведениях — не ломай существующую логику.

Стек: NestJS + Prisma + PostgreSQL + Socket.IO + React 18 + Tailwind CSS + Zustand + TanStack Query.

Исходный код находится локально. Перед написанием любого нового кода — открой соответствующий
существующий модуль и изучи его структуру.

---

## Законодательный контекст (Кыргызстан)

- По статье 109 Налогового кодекса КР — ККМ обязательна при наличных и безналичных расчётах с населением
- С 1 сентября 2016 года все ККМ обязаны передавать данные в ГНС (Государственную налоговую службу)
- ККМ регистрируется через ЦТО (Центр технического обслуживания), не через налоговую напрямую
- ККМ имеет секции: секция 1 — наличные, секция 2 — безналичные (карта/QR)
- Данные передаются в ГНС через УПФД (устройство передачи фискальных данных)

## Провайдеры ККМ в Кыргызстане

### 1. eKassa — приоритетный провайдер
- Провайдер: Telemedia Group (ekassa.kg)
- Тип: облачная онлайн-ККМ с REST API
- Стоимость: 500 сом/месяц (владелец платит напрямую Telemedia Group)
- Уже интегрирована с: rKeeper, 1С, Glovo, UMAG, CDEK и другими
- Внесена в государственный реестр ККМ КР (постановление КМ КР №193 от 08.04.2022)
- Автоматическая передача данных в ГНС при формировании чека
- Подключается за 1 день из любой точки Кыргызстана
- Поддерживает: наличные, карта, QR — через отдельные секции ККМ
- Документация API: запросить у Telemedia Group (ekassa.kg, WhatsApp: +996 555 979 242)

### 2. YaKassa — альтернативный провайдер
- Провайдер: YaKassa (yakassa.kg)
- Тип: облачная онлайн-ККМ с API
- Внесена в реестр ЦТО ГНС КР
- Альтернатива если владелец уже использует YaKassa
- Документация API: запросить у YaKassa напрямую

## Цель задачи

Добавить интеграцию с ККМ. Владелец выбирает провайдера в настройках — **eKassa** (приоритет)
или **YaKassa** (альтернатива). Без настроенного провайдера печать фискального чека недоступна.

---

## Схема работы

Один термопринтер выполняет разные функции в зависимости от этапа и настроек.

### Без ККМ (текущая схема)

```
1. Счёт (до оплаты)
   └── Термопринтер → предварительный счёт клиенту

2. Клиент соглашается и платит

3. Администратор принимает оплату в системе

4. Товарный чек (после оплаты)
   └── Термопринтер → товарный чек клиенту
```

### С ККМ включённой (новая схема)

```
1. Счёт (до оплаты)
   └── Термопринтер → предварительный счёт клиенту  ← без изменений

2. Клиент соглашается и платит

3. Администратор принимает оплату в системе

4. Фискальный чек (после оплаты)
   ├── Товарный чек НЕ печатается (отключается)
   ├── Запрос в eKassa API → данные уходят в ГНС автоматически
   └── Термопринтер → фискальный чек  ← тот же принтер, тот же момент
```

Фискальный чек содержит всё то же что товарный (позиции, суммы, название заведения)
плюс QR-код ГНС и фискальный номер. Клиент получает ровно один чек после оплаты —
просто теперь он официальный с юридической силой.

**Оборудование не меняется.** Новый принтер покупать не нужно.
Владелец подключает eKassa самостоятельно (500 сом/мес напрямую Telemedia Group),
вводит API-ключ в настройках — и система переключается на фискальные чеки.

ККМ встраивается только в **один момент** — когда администратор подтверждает печать
(статус `approved` → `printed`). Это уже существующая точка в коде.

## Что НЕ трогать

- Существующий workflow печати чека (`receipt-prints` модуль) — не переписывать
- Логику официанта — не трогать вообще
- Логику оплаты (`payments`) — не трогать
- Термопринтер — продолжает работать как раньше
- Если провайдер не выбран — кнопка «Печать фискального чека» неактивна, администратор видит подсказку настроить ККМ в настройках

## Что реализовать

### 1. Backend — новый модуль `fiscal/`

```
backend/src/fiscal/
├── fiscal.module.ts
├── fiscal.service.ts              — выбирает провайдера по настройкам
├── fiscal.interface.ts            — интерфейс IFiscalProvider
├── providers/
│   ├── ekassa.provider.ts         — интеграция с eKassa (приоритет)
│   └── yakassa.provider.ts        — интеграция с YaKassa (альтернатива)
└── dto/
    └── fiscal-receipt.dto.ts
```

**Интерфейс:**
```typescript
interface IFiscalProvider {
  printReceipt(data: FiscalReceiptData): Promise<FiscalResult>
  testConnection(): Promise<boolean>  // для кнопки «Проверить соединение»
}

interface FiscalReceiptData {
  orderId: number
  items: Array<{
    name: string       // dishSnapshot из заказа (не из текущего блюда!)
    price: number      // priceSnapshot из заказа
    quantity: number
  }>
  totalAmount: number
  paymentType: 'cash' | 'card' | 'qr' | 'mixed'
  cashAmount?: number  // если наличные — сколько внесено (для сдачи)
  section?: number     // секция ККМ: 1 — наличные, 2 — безналичные
}

interface FiscalResult {
  success: boolean
  fiscalReceiptNumber?: string  // номер фискального чека
  fiscalSign?: string           // фискальный признак
  qrCode?: string               // QR для проверки
  error?: string
}
```

**eKassaProvider** — интеграция с eKassa API:
- Базовый URL: получить из документации eKassa (ekassa.kg/api)
- Аутентификация: API-ключ в заголовке (уточнить у Telemedia Group)
- При оплате наличными — передавать секцию 1
- При оплате картой/QR — передавать секцию 2
- При смешанной оплате (mixed) — разбить на два запроса по секциям

**FiscalService:**
```typescript
async getProvider(): Promise<IFiscalProvider> {
  const settings = await this.settingsService.get()
  switch (settings.fiscalProvider) {
    case 'ekassa': return new EkassaProvider(settings.fiscalEkassaConfig)
    case 'yakassa': return new YakassaProvider(settings.fiscalYakassaConfig)
    default: throw new Error('Провайдер ККМ не настроен. Выберите eKassa или YaKassa в настройках.')
  }
}
```

### 2. Встройка в существующий код

Найти в `backend/src/receipt-prints/` метод подтверждения печати.
Добавить **одну строку** вызова fiscalService:

```typescript
async approveReceiptPrint(id: number) {
  // ... существующая логика без изменений ...

  // ДОБАВИТЬ: фискальный чек если ККМ подключён
  const settings = await this.settingsService.get()
  const fiscalResult = await this.fiscalService.printReceipt({
    orderId: receiptPrint.orderId,
    items: receiptPrint.order.items.map(i => ({
      name: i.dishSnapshot,      // snapshot — не брать из dish напрямую
      price: i.priceSnapshot,
      quantity: i.quantity,
    })),
    totalAmount: receiptPrint.order.totalAmount,
    paymentType: receiptPrint.order.payment.type,
    cashAmount: receiptPrint.order.payment.cashGiven,
    // Секция: 1 для наличных, 2 для карты/QR
    section: receiptPrint.order.payment.type === 'cash' ? 1 : 2,
  })

  // Сохранить результат
  await this.prisma.receiptPrint.update({
    where: { id },
    data: {
      fiscalReceiptNumber: fiscalResult.fiscalReceiptNumber ?? null,
      fiscalSign: fiscalResult.fiscalSign ?? null,
      fiscalQrCode: fiscalResult.qrCode ?? null,
      fiscalError: fiscalResult.error ?? null,
    }
  })

  // Ошибка ККМ не блокирует flow — только уведомление
  // ... остальная существующая логика без изменений ...
}
```

### 4. Frontend — блок настроек ККМ

В существующем разделе настроек администратора добавить блок **«ККМ / Фискализация»**:

```
┌─────────────────────────────────────────────────┐
│  ККМ / Фискализация                             │
│                                                 │
│  Провайдер                                      │
│  ○ Выключено                                    │
│  ○ eKassa (Telemedia Group)                     │
│  ○ YaKassa                                      │
│                                                 │
│  [если выбран eKassa:]                          │
│  API-ключ        [__________________________]   │
│  ИНН заведения   [__________________________]   │
│                                                 │
│  [если выбран YaKassa:]                         │
│  API-ключ        [__________________________]   │
│                                                 │
│  [Проверить соединение]                         │
│  ✅ Соединение установлено                      │
└─────────────────────────────────────────────────┘
```

- Автосохранение (как в остальных настройках edu-pos — без кнопки «Сохранить»)
- Кнопка «Проверить соединение» вызывает `fiscalService.testConnection()`
- Поля показываются/скрываются в зависимости от выбранного провайдера

### 5. Frontend — карточка заказа

В карточке заказа (детали для администратора) добавить:

- **ККМ выключен** → ничего не показывать
- **ККМ включён, чек пробит** → показать номер фискального чека + QR (если есть)
- **Ошибка ККМ** → красный бейдж «Ошибка ККМ» + текст ошибки + кнопка «Повторить»

---

### 6. Обработка ошибок

Ошибка ККМ **не блокирует** основной flow:

- Записать ошибку в `ReceiptPrint.fiscalError`
- Toast администратору: «Чек пробит, но ККМ вернул ошибку. Проверьте подключение.»
- Статус чека (`printed`) выставить — не откатывать
- Кнопка «Повторить фискализацию» в карточке заказа — повторный вызов fiscalService
